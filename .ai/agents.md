# Agent Roles

## 한 줄 원칙

**gemini-researcher는 초안을 작성하고, codex-reasoner는 현재 코드 기준으로 검증하고, Claude는 확정된 내용만 구현한다. 확정되지 않은 초안은 바로 코드로 옮기지 않는다.**

---

## 판단 vs 질문 — 언제 사람에게 묻는가

### 스스로 판단하고 진행하는 경우

아래 소스로 답을 찾을 수 있으면 사람에게 묻지 않고 진행한다:

| 소스 | 확인 내용 |
|---|---|
| `.ai/docs/PROJECT.md` | 현재 정책·런타임 계약 |
| `.ai/docs/TODO.md` | 우선순위·미완료 항목 |
| `.ai/docs/API_DOCS.md` | API 계약 |
| `CURRENT_TASK.md` | 현재 진행 중인 작업·설계 확정 내용 |
| 실제 코드 | 현재 구현 현황 |
| `gemini-researcher` 호출 | 공식 문서·외부 스펙 확인 |
| `codex-reasoner` 호출 | 복잡한 로직·보안 분석 |

### 반드시 사람에게 묻는 경우

다음 중 하나라도 해당하면 **구현을 멈추고 사람에게 확인**한다:

1. **정책 미확정** — 문서 어디에도 명확한 정책이 없을 때
2. **설계 충돌** — PROJECT.md / CURRENT_TASK.md / 실제 코드가 서로 다른 말을 할 때
3. **운영 리스크** — 프로덕션 데이터 변경, 인증 키 관련 작업
4. **범위 불명확** — 요청이 어디까지인지 해석이 두 가지 이상 가능할 때
5. **미정 항목** — spec에 `미정 항목`으로 명시된 내용에 대한 결정이 필요할 때

### 질문 형식

사람에게 물을 때는 아래 형식으로 간결하게:

```
[확인 필요]
- 상황: (무엇을 하려다 막혔는가)
- 모호한 점: (어떤 부분이 불명확한가)
- 옵션: A / B (가능한 선택지가 있다면)
- 근거 문서: (확인한 문서 경로)
```

## 라우팅 판단 기준

| 질문 | 담당 |
|---|---|
| 복잡한 알고리즘, 비즈니스 로직 검증, 보안 취약점 분석이 필요한가? | `codex-reasoner` |
| 최신 공식 문서, 라이브러리 버전, 외부 정보가 필요한가? | `gemini-researcher` |
| 기능 기획 초안 작성이 필요한가? | `gemini-researcher` |
| 전체 흐름 설계, 코드 생성·수정, 결과 통합이 필요한가? | Claude |

---

## Claude (오케스트레이터)

**역할**
- 작업 흐름 전체 설계 및 조율
- 코드 생성·수정·리팩터링
- gemini-researcher 초안 + codex-reasoner 리뷰 결과 통합
- 확정된 내용만 구현
- 구현 후 관련 문서 동기화
- CURRENT_TASK.md 업데이트

**하지 말 것**
- 복잡한 로직 수학적 검증 → codex-reasoner에 위임
- 최신 라이브러리 API·버전 확인 → gemini-researcher에 위임
- `.env` 파일 읽기 (절대 금지)
- 확정되지 않은 기획 초안 바로 구현

---

## codex-reasoner (깊은 추론)

**투입 시점**
- 복잡한 비즈니스 로직 검증
- 보안 취약점 분석
- 버그 근본 원인 추론 (단계별 깊은 분석)
- gemini-researcher 초안이 현재 코드/문서와 맞는지 검토

**Codex에게 전달할 때 (Claude가 정리)**
```
## 작업 목표
[무엇을 검증/분석해야 하는지]

## 배경
[왜 이 분석이 필요한지]

## 관련 코드
[파일 경로 또는 스니펫]

## 기대 결과물
[단계별 추론 / 취약점 목록 / 수정 방향]
```

---

## gemini-researcher (리서처 + 기획 초안)

**투입 시점**
- 공식 문서 확인, 최신 버전 breaking change 조회
- 외부 서비스 스펙 확인
- 신규 기능 기획 초안 작성 (제안서 역할, 구현 기준 아님)

**Gemini에게 전달할 때 (Claude가 정리)**
```
## 확인 질문 또는 기획 요청
[구체적인 질문 또는 기능명]

## 현재 시스템 기준
[현재 사용 중인 방식 또는 관련 문서 항목]

## 배경
[왜 확인/기획이 필요한지]

## 기대 결과물
[공식 문서 인용 / 최신 스펙 / 기획 초안]
```

---

## 기능 개발 워크플로우

```
1. gemini-researcher → 기능 기획 초안 작성
   (목적 / 사용자 시나리오 / 정책 / API 계약 초안 / 실패 케이스 / 미정 항목)

2. codex-reasoner   → 현재 코드/문서 기준 리뷰
   (맞는 점 / 틀린 점 / 빠진 점 / 위험한 점)

3. 사람             → 수정·확정

4. Claude           → 확정된 내용만 구현
                    → 구현 후 관련 문서 동기화
                    → CURRENT_TASK.md 업데이트
                    → draft 파일 삭제
```

## 문서별 역할

| 문서 | 내용 기준 |
|---|---|
| `.ai/docs/PROJECT.md` | 현재 상태만 (아키텍처 경계, 정책, 런타임 계약) |
| `.ai/docs/TODO.md` | 할 일만 (미완료·보류·우선순위) |
| `.ai/docs/API_DOCS.md` | 외부 요청/응답 계약 |
| `.ai/docs/specs/*.draft.md` | gemini-researcher 초안 (임시, 구현 완료 시 삭제) |
| `.ai/docs/specs/*.review.md` | codex-reasoner 리뷰 (임시, 구현 완료 시 삭제) |
| `CURRENT_TASK.md` | 현재 작업 상태 + AI 핸드오프 노트 |
